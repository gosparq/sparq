## Overview

sparQ uses a **module-scoped translation system**:

- JSON files in each module's `lang/` directory
- Module-specific translations with fallback to core
- Loaded at startup into memory for fast access
- `{{ _("text") }}` syntax in templates
- Supports date/number formatting per locale

**Fallback Chain:**
1. Current module's translation
2. Core module's translation
3. Original text (unchanged)

**Key Insight:** Since English text passes through unchanged, `en.json` is **optional**. Only create it for runtime string overrides.

### Why JSON, Not gettext?

Python's standard i18n path is gettext with Flask-Babel. sparQ deliberately chose JSON files instead. Here's why.

**No toolchain.** gettext requires `xgettext` to extract strings, `msgfmt` to compile `.po` → `.mo`, and `pybabel` as a wrapper. That's external dependencies and a build step. JSON files just work — no extraction, no compilation, no build pipeline.

**No binary artifacts.** gettext compiles `.po` into `.mo` binary files. Forget to compile and translations silently fail. `.mo` files can't be diffed or reviewed in PRs. JSON loads directly and diffs cleanly.

**Module-scoped by default.** gettext uses a single message catalog (or one domain per module with manual wiring). sparQ's approach naturally scopes translations per module folder — each module owns its `lang/` directory, no catalog merging needed.

**Translators can edit directly.** JSON is something any translator or non-developer can open, edit, and validate. `.po` files have headers, fuzzy markers, and `msgid`/`msgstr` syntax that's confusing outside of developer tooling.

**Runtime extensibility.** Language packs can be installed and uninstalled at runtime via the Control Panel. gettext would require recompilation and app restart.

**English as the key.** Using the actual English string as the translation key means templates read naturally: `_("Save Changes")` not `_("btn_save_changes")`. gettext supports this via `msgid`, but wraps it in `.po` ceremony.

**When you'd reconsider.** If you need plural forms (`ngettext`), gender agreement, or ICU MessageFormat, a heavier system pays for itself. sparQ sidesteps this by using template-level branching for the few cases where it matters.

## Translation File Structure

### Location

Each module can have its own translations:

```
modules/
├── core/
│   └── lang/
│       ├── en.json
│       ├── es.json
│       └── nl.json
├── team/
│   └── lang/
│       ├── en.json
│       └── es.json
└── yourmodule/
    └── lang/
        ├── en.json
        └── es.json
```

### JSON Format

```json
// data/modules/apps/yourapp/lang/en.json
{
    "Items": "Items",
    "Add Item": "Add Item",
    "Item created": "Item created",
    "Delete": "Delete",
    "Are you sure?": "Are you sure?"
}
```

```json
// data/modules/apps/yourapp/lang/es.json
{
    "Items": "Artículos",
    "Add Item": "Agregar Artículo",
    "Item created": "Artículo creado",
    "Delete": "Eliminar",
    "Are you sure?": "¿Estás seguro?"
}
```

### Metadata for Formatting

Use `_meta` key for locale-specific formatting:

```json
// modules/core/lang/en.json
{
    "_meta": {
        "date_formats": {
            "short": "MM/DD/YYYY",
            "medium": "YYYY-MM-DD",
            "long": "YYYY-MM-DD HH:mm"
        },
        "number_formats": {
            "decimal_separator": ".",
            "thousand_separator": ","
        }
    },
    "Login": "Login",
    "Logout": "Logout"
}
```

```json
// modules/core/lang/es.json
{
    "_meta": {
        "date_formats": {
            "short": "DD/MM/YYYY",
            "medium": "DD-MM-YYYY",
            "long": "DD-MM-YYYY HH:mm"
        },
        "number_formats": {
            "decimal_separator": ",",
            "thousand_separator": "."
        }
    },
    "Login": "Iniciar sesión",
    "Logout": "Cerrar sesión"
}
```

## Translation System

### System Implementation

```python
# system/i18n/translation.py
import json
import os
from flask import g, current_app

TRANSLATIONS = {}


def preload_translations():
    """Load all translations at startup."""
    global TRANSLATIONS
    TRANSLATIONS = {}

    modules_path = os.path.join(current_app.root_path, "modules")

    for module_name in os.listdir(modules_path):
        module_path = os.path.join(modules_path, module_name)
        if not os.path.isdir(module_path):
            continue

        lang_path = os.path.join(module_path, "lang")
        if os.path.isdir(lang_path):
            for lang_file in os.listdir(lang_path):
                if lang_file.endswith(".json"):
                    lang_code = lang_file.replace(".json", "")
                    file_path = os.path.join(lang_path, lang_file)

                    with open(file_path, "r", encoding="utf-8") as f:
                        TRANSLATIONS.setdefault(lang_code, {}).setdefault(
                            module_name, {}
                        ).update(json.load(f))


def translate(text):
    """Translate text with module fallback."""
    lang = g.get("lang", current_app.config.get("DEFAULT_LANGUAGE", "en"))
    current_module = g.get("current_module", {}).get("name", "core").lower()

    # Try module-specific translation
    if current_module != "core":
        result = TRANSLATIONS.get(lang, {}).get(current_module, {}).get(text)
        if result:
            return result

    # Fallback to core
    result = TRANSLATIONS.get(lang, {}).get("core", {}).get(text)
    if result:
        return result

    # Return original text
    return text


# Alias
_ = translate
```

### Registration in app.py

```python
# In app.py
from system.i18n.translation import translate, format_date, format_number, preload_translations

# Add to Jinja globals
app.jinja_env.globals["_"] = translate
app.jinja_env.filters["format_date"] = format_date
app.jinja_env.filters["format_number"] = format_number

# Load translations
with app.app_context():
    preload_translations()
```

## Date and Number Formatting

### Date Filter

```python
# system/i18n/translation.py
def format_date(date, format_type="medium"):
    """Format date according to locale."""
    if not date:
        return ""

    patterns = get_format_patterns()
    date_formats = patterns.get("date_formats", {})
    pattern = date_formats.get(format_type, "%Y-%m-%d")

    # Convert to strftime format
    pattern = (
        pattern.replace("YYYY", "%Y")
        .replace("MM", "%m")
        .replace("DD", "%d")
        .replace("HH", "%H")
        .replace("mm", "%M")
    )

    if isinstance(date, str):
        from datetime import datetime
        date = datetime.strptime(date, "%Y-%m-%d")

    return date.strftime(pattern)
```

### Number Filter

```python
def format_number(number, decimal_places=2):
    """Format number according to locale."""
    patterns = get_format_patterns()
    formats = patterns.get("number_formats", {})
    decimal_sep = formats.get("decimal_separator", ".")
    thousand_sep = formats.get("thousand_separator", ",")

    number_str = f"{number:,.{decimal_places}f}"
    if thousand_sep != ",":
        number_str = number_str.replace(",", thousand_sep)
    if decimal_sep != ".":
        number_str = number_str.replace(".", decimal_sep)

    return number_str
```

### Template Usage

```html
<!-- Date formatting -->
<td>{{ item.created_at|format_date("short") }}</td>
<td>{{ item.created_at|format_date("medium") }}</td>
<td>{{ item.created_at|format_date("long") }}</td>

<!-- Number formatting -->
<td>{{ item.price|format_number(2) }}</td>
<td>{{ item.quantity|format_number(0) }}</td>
```

## Language Detection

### Detection Chain

Language is determined in this order:

1. **URL parameter**: `?lang=es`
2. **Session**: Previously selected language
3. **User setting**: From database (if logged in)
4. **Default**: `en`

### Implementation in app.py

```python
@app.before_request
def before_request():
    # Language detection
    g.lang = (
        request.args.get("lang")
        or session.get("lang")
        or (
            UserSetting.get(current_user.id, "language")
            if current_user.is_authenticated
            else None
        )
        or app.config.get("DEFAULT_LANGUAGE", "en")
    )

    # Persist to session
    if session.get("lang") != g.lang:
        session["lang"] = g.lang
```

### Language Switcher

```html
<!-- In template -->
<div class="dropdown">
    <button class="btn btn-outline-secondary dropdown-toggle" data-bs-toggle="dropdown">
        {{ g.lang|upper }}
    </button>
    <ul class="dropdown-menu">
        <li><a class="dropdown-item" href="?lang=en">English</a></li>
        <li><a class="dropdown-item" href="?lang=es">Español</a></li>
        <li><a class="dropdown-item" href="?lang=nl">Nederlands</a></li>
    </ul>
</div>
```

## Adding Translations

### To a New Module

1. Create the `lang/` directory:
```bash
mkdir -p data/modules/apps/yourapp/lang
```

2. Add non-English languages:
```json
// data/modules/apps/yourapp/lang/es.json
{
    "Module Title": "Título del Módulo",
    "Welcome": "Bienvenido",
    "Save": "Guardar"
}
```

3. Use in templates:
```html
<h1>{{ _("Module Title") }}</h1>
```

**Note:** `en.json` is **optional**. English strings pass through unchanged when no translation is found (see Fallback Chain above). Only create `en.json` if you need to override specific English strings at runtime.

### English Overrides (Optional)

Create `en.json` only when you need to override specific English strings:

```json
// data/modules/apps/yourapp/lang/en.json (sparse - only overrides)
{
    "Items": "Products"
}
```

Use cases for `en.json`:
- **Terminology changes**: Override "Items" → "Products" for a specific module
- **Runtime modifications**: Change labels without code deployment
- **A/B testing**: Test different copy variations

If no override is needed, skip creating `en.json` entirely.

### To Core Module

Add translations that should be available everywhere:

```json
// modules/core/lang/en.json
{
    "Save": "Save",
    "Cancel": "Cancel",
    "Delete": "Delete",
    "Edit": "Edit",
    "Loading...": "Loading...",
    "Error": "Error",
    "Success": "Success"
}
```

These are fallbacks for all modules.

## Best Practices

### Keys

✅ **Do:**
- Use the actual English text as the key
- Keep keys descriptive and complete
- Use same key for same meaning

```json
{
    "Save Changes": "Save Changes",
    "Add New Item": "Add New Item"
}
```

❌ **Don't:**
- Don't use abbreviations or codes
- Don't use different keys for same text

```json
{
    "btn_save": "Save",  // Bad: Use "Save" instead
    "save_btn": "Save"   // Bad: Duplicate
}
```

### Translations

✅ **Do:**
- Provide complete translations for non-English languages
- Test formatting in context
- Consider text length variations
- Use sparse `en.json` (only overrides, or skip entirely)

❌ **Don't:**
- Don't leave empty strings (remove key instead)
- Don't machine-translate without review
- Don't assume word order is same
- Don't duplicate English strings in `en.json` (they pass through unchanged)

### Module Scoping

✅ **Do:**
- Put module-specific text in module's lang folder
- Put shared text in core module
- Let module translations override core

❌ **Don't:**
- Don't duplicate translations across modules
- Don't put module-specific text in core

### Template Discipline

✅ **Do:**
```html
<button>{{ _("Save") }}</button>
<h1>{{ _("Dashboard") }}</h1>
```

❌ **Don't:**
```html
<button>Save</button>  <!-- Bad: Hardcoded -->
<h1>Dashboard</h1>     <!-- Bad: Hardcoded -->
```

## Troubleshooting

### Translation Not Found

1. Check file exists: `data/modules/apps/yourapp/lang/en.json`
2. Check JSON is valid (no trailing commas)
3. Check key matches exactly (case-sensitive)
4. Restart app to reload translations

### Module Translation Not Used

1. Verify `g.current_module` is set correctly
2. Check module name matches folder name (lowercase)
3. Translations fall back to core if module's is missing

### Formatting Issues

1. Check `_meta` section in JSON
2. Verify format patterns are correct
3. Check date/number input types
